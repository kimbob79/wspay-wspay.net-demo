# 개발 이력 - 2026-02-21

## 수기결제 NOTI 상태 체크 기능 추가 및 금액 조건 매칭 개선

### 수정 파일
- `manual_payment.php`

### 작업 배경
- 수기결제(g5_payment_keyin) 승인 건이 PG사 NOTI를 정상적으로 수신했는지, g5_payment에 등록되었는지 확인하는 기능이 필요했음
- 초기 구현 시 승인번호(appNo)만으로 NOTI 매칭 → 동일 승인번호가 다른 거래에서 재사용될 수 있는 문제
- 금액도 함께 비교하는 복합키 방식으로 개선하여 매칭 정확도 향상

### 2. NOTI 필터 SQL - EXISTS 서브쿼리 (3개 필터 × 6개 PG)

**위치**: PHP 상단 SQL 생성 로직 (line 128~211)

승인 건(`pk_status='approved'`)만 대상으로, 3가지 필터 조건 구현:

#### 2-1. `noti_missing` (NOTI 미수신)
PG NOTI 테이블에 해당 승인번호+금액 조합의 레코드가 없는 경우

```sql
-- paysis
NOT EXISTS (SELECT 1 FROM g5_payment_paysis noti
    WHERE noti.connCd='0005' AND noti.appNo = p.pk_app_no
    AND CAST(noti.amt AS SIGNED) = p.pk_amount)

-- stn
NOT EXISTS (SELECT 1 FROM g5_payment_stn noti
    WHERE noti.requestFlag='K' AND noti.applNo = p.pk_app_no
    AND CAST(noti.amount AS SIGNED) = p.pk_amount)

-- rootup
NOT EXISTS (SELECT 1 FROM g5_payment_routeup noti
    WHERE noti.module_type='1' AND noti.appr_num = p.pk_app_no
    AND CAST(noti.amount AS SIGNED) = p.pk_amount)

-- winglobal (daou + korpay + danal 모두 NOT EXISTS)
NOT EXISTS (SELECT 1 FROM g5_payment_daou noti
    WHERE noti.appNo = p.pk_app_no AND CAST(noti.amt AS SIGNED) = p.pk_amount)
AND NOT EXISTS (SELECT 1 FROM g5_payment_korpay noti
    WHERE noti.appNo = p.pk_app_no AND CAST(noti.amt AS SIGNED) = p.pk_amount)
AND NOT EXISTS (SELECT 1 FROM g5_payment_danal noti
    WHERE noti.CARDAUTHNO = p.pk_app_no AND CAST(noti.AMOUNT AS SIGNED) = p.pk_amount)
```

#### 2-2. `payment_missing` (미등록)
PG NOTI는 존재하지만 g5_payment 메인 테이블에 등록되지 않은 경우

```sql
-- paysis/stn/rootup: NOTI EXISTS + g5_payment NOT EXISTS
EXISTS (NOTI 조회...) AND NOT EXISTS (
    SELECT 1 FROM g5_payment gp
    WHERE gp.pg_name IN ('paysis_keyin','stn_k','routeup_k')
    AND gp.pay_num = p.pk_app_no AND gp.pay = p.pk_amount
)

-- winglobal: NOTI EXISTS (daou OR korpay OR danal) + g5_payment NOT EXISTS
(EXISTS daou OR EXISTS korpay OR EXISTS danal) AND NOT EXISTS (
    SELECT 1 FROM g5_payment gp
    WHERE gp.pg_name IN ('daou','korpay','danal')
    AND gp.pay_num = p.pk_app_no AND gp.pay = p.pk_amount
)
```

#### 2-3. `normal` (정상)
PG NOTI + g5_payment 모두 존재하는 경우

```sql
-- payment_missing과 동일 구조, NOT EXISTS → EXISTS로 변경
EXISTS (NOTI 조회...) AND EXISTS (g5_payment 조회...)
```

#### 금액 비교 핵심 포인트
- PG NOTI 테이블의 금액 컬럼이 **varchar** 타입 → `CAST(noti.amt AS SIGNED)` 로 int 변환 후 비교
- g5_payment의 `pay` 컬럼은 int → 직접 비교 `gp.pay = p.pk_amount`

### 3. 배치 NOTI 상태 체크 로직 (관리자 전용)

**위치**: PHP 상단, SQL 결과 조회 후 (line 275~349)

목록 조회 결과를 배열로 수집한 뒤, 승인번호+금액 복합키로 NOTI 존재 여부를 배치 체크:

#### 3-1. 결과 배열 수집
```php
// 관리자: sql_fetch_array 루프 → $rows_data 배열로 수집
// 가맹점: 기존대로 for 루프에서 직접 출력
$rows_data = array();
while($r = sql_fetch_array($result)) {
    $rows_data[] = $r;
}
```

#### 3-2. 승인번호를 PG별로 분류
```php
$pg_app_nos = array('paysis' => [], 'stn' => [], 'rootup' => [], 'winglobal' => []);
$all_app_nos = array();
foreach($rows_data as $r) {
    if($r['pk_status'] == 'approved' && $r['pk_app_no'] && isset($pg_app_nos[$r['pk_pg_code']])) {
        $pg_app_nos[$r['pk_pg_code']][] = "'" . sql_escape_string($r['pk_app_no']) . "'";
        $all_app_nos[] = "'" . sql_escape_string($r['pk_app_no']) . "'";
    }
}
```

#### 3-3. PG별 NOTI 존재 체크 (복합키)

| PG | 테이블 | SELECT 컬럼 | WHERE 조건 | 복합키 |
|---|---|---|---|---|
| paysis | g5_payment_paysis | `appNo, amt` | `connCd='0005'` | `appNo_amt` |
| stn | g5_payment_stn | `applNo, amount` | `requestFlag='K'` | `applNo_amount` |
| rootup | g5_payment_routeup | `appr_num, amount` | `module_type='1'` | `appr_num_amount` |
| winglobal-daou | g5_payment_daou | `appNo, amt` | - | `appNo_amt` |
| winglobal-korpay | g5_payment_korpay | `appNo, amt` | - | `appNo_amt` |
| winglobal-danal | g5_payment_danal | `CARDAUTHNO, AMOUNT` | - | `CARDAUTHNO_AMOUNT` |

```php
// 예시: paysis
$q = sql_query("SELECT appNo, amt FROM g5_payment_paysis WHERE connCd='0005' AND appNo IN ({$in})");
while($nr = sql_fetch_array($q)) {
    $noti_found[$nr['appNo'].'_'.$nr['amt']] = true;
}
```

#### 3-4. g5_payment 존재 체크 (복합키)
```php
$q = sql_query("SELECT pay_num, pay FROM g5_payment
    WHERE pg_name IN ('paysis_keyin','stn_k','routeup_k','daou','korpay','danal')
    AND pay_num IN ({$in})");
while($pr = sql_fetch_array($q)) {
    $payment_found[$pr['pay_num'].'_'.$pr['pay']] = true;
}
```

#### 3-5. 상태 맵 생성
```php
foreach($rows_data as $r) {
    $app = $r['pk_app_no'];
    $key = $app . '_' . $r['pk_amount'];  // 복합키
    if($r['pk_status'] == 'approved' && $app && in_array($r['pk_pg_code'], ['paysis','stn','rootup','winglobal'])) {
        $has_noti = isset($noti_found[$key]);
        $has_payment = isset($payment_found[$key]);
        if($has_noti && $has_payment) {
            $noti_check[$key] = 'normal';        // 정상
        } else if(!$has_noti) {
            $noti_check[$key] = 'noti_missing';   // NOTI 미수신
        } else {
            $noti_check[$key] = 'payment_missing'; // 미등록
        }
    }
}
```

### 4. NOTI 컬럼 및 뱃지 표시 (관리자 테이블)

**위치**: 관리자용 테이블 thead + tbody

#### 4-1. 테이블 헤더
- `<th>NOTI</th>` 컬럼 추가 (PG 컬럼 다음)
- `colspan` 17 → 18 변경

#### 4-2. 행 루프 변경
- 기존: `for ($i=0; $row=sql_fetch_array($result); $i++)`
- 변경: `foreach($rows_data as $i => $row)` (배치 체크 때 수집한 배열 사용)

#### 4-3. 뱃지 표시 로직
```php
if($row['pk_status'] == 'approved' && $row['pk_app_no']
   && in_array($row['pk_pg_code'], ['paysis','stn','rootup','winglobal'])) {
    $noti_key = $row['pk_app_no'] . '_' . $row['pk_amount'];  // 복합키
    $noti_st = isset($noti_check[$noti_key]) ? $noti_check[$noti_key] : '';
    if($noti_st == 'normal') {
        echo '<span class="noti-badge noti-ok">정상</span>';
    } else if($noti_st == 'noti_missing') {
        echo '<span class="noti-badge noti-miss">NOTI미수신</span>';
    } else if($noti_st == 'payment_missing') {
        echo '<span class="noti-badge noti-unreg">미등록</span>';
    }
} else {
    echo '-';
}
```

## NOTI 상태 판정 로직 요약

```
승인 건 (pk_status='approved', pk_app_no 존재)에 대해:

복합키 = 승인번호 + '_' + 금액

1. PG NOTI 테이블에 복합키 매칭 레코드 있음?
   → NO: "NOTI미수신" (빨간 뱃지)
   → YES:
     2. g5_payment 테이블에 복합키 매칭 레코드 있음?
        → NO: "미등록" (주황 뱃지)
        → YES: "정상" (초록 뱃지)
```
